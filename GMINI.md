# CLAUDE.md — Agent Reliability Operating Protocol

> ⛔ 이 문서의 모든 지시는 **MANDATORY(필수)**다. 선택 사항 없음. 단 하나도 건너뛰지 마라.
> **어떤 작업이든 — 코드·문서·분석·설계·리뷰 — 이 프로토콜을 적용한다.**
> **모든 확인은 증거 기반이어야 한다. 직접 열어 1줄씩 읽는 것만 인정.**

---

## ══ ABSOLUTE PROHIBITIONS — 절대 금지 ══

> 아래를 위반하면 작업 자체가 무효다. 이유 불문.

```
❌ grep / 패턴 매칭 / 기억만으로 "확인했다" 판단
   → 반드시 관련 파일을 직접 열어 해당 줄을 눈으로 읽어야 한다

❌ 가정(assumption) 기반 수정
   → "아마 이렇게 되어 있을 것이다"로 코드를 건드리는 행위 전면 금지

❌ 할루시네이션 언어 사용
   → "probably" "should work" "I think" "likely" "아마" "대충" 금지
   → 모르면 반드시 "I don't know — 파일을 열어 확인하겠습니다" 선언

❌ 확인 안 된 API·함수·파일·변수·경로 참조
   → 파일을 직접 열어 존재와 시그니처를 확인한 것만 참조 가능

❌ 피상적 체크
   → 체크박스를 체크하려면 시작 전·완료 후 각각 직접 추적이 필요하다
   → 추적 근거(파일명, 확인 내용)를 기록해야 한다

❌ Post-Work 전수조사 생략·축약·순서 변경

❌ 범위 선언 없이 수정 — 먼저 "무엇을 / 어디서 / 왜"를 선언한다

❌ 작업 완료 후 영향받은 것(테스트·타입·상수·문서·import) 싱크 누락

❌ 리팩토링 + 기능 추가 동일 커밋

❌ 구 코드를 사용 안 하면서 삭제하지 않고 방치

❌ 세션 종료 시 메모리 추출 생략 — 반드시 SESSION MEMORY PROTOCOL 실행
   → 다음 세션이 컨텍스트 없이 시작하는 것은 비용 낭비이자 리스크다

❌ 병렬 에이전트 간 충돌 사전 점검 없이 작업 착수
   → 동일 파일을 건드리는 에이전트가 있는지 먼저 확인한다
```

## ══ UNIVERSAL INVESTIGATION PROTOCOL — 모든 작업에 공통 적용 ══

> **작업 유형과 무관하게** — 코드 수정, 기능 개발, 문서 작성, 분석, 설계 검토, 버그 수정 — 아래를 반드시 따른다.

### 작업 시작 전 (Before Any Work)

```
1. 관련된 모든 파일을 직접 열어 처음부터 끝까지 1줄씩 읽는다.
   → 코드:    소스 파일 전체
   → 문서:    기존 문서 + 연관 코드/API 전체
   → 분석:    분석 대상 파일 전체
   → 설계:    현행 구조 파일 전체

2. 현행 상태를 완전히 파악한다 — thoroughly.
   → 진입점 / 데이터 흐름 / 의존성 / 동적 연결 / 공개 API 목록

3. 영향받는 모든 파일 목록을 작성한다.

4. 범위를 선언한다:
   - 무엇을(Target):
   - 왜(Reason):
   - 어디까지(Scope):
   - 예상 영향(Impact):
   - 롤백 방법(Rollback):

5. 작업 전 상태가 안정적임을 확인한다.
   - 코드 작업: 빌드 성공 · 테스트 100% pass
   - 문서 작업: 기존 문서의 현행 반영 상태 확인
   - 분석 작업: 분석 기준선 데이터 확보

→ 위 5가지를 완료한 뒤에만 작업을 시작한다.
```

### 체크박스 단위 규칙 (Per-Checkbox — 예외 없음)

모든 체크박스는 다음 구조로 실행된다:

```
[ □ 체크박스: <내용> ]

  [시작 전 — 이것을 완료하기 전에는 체크박스를 시작하지 않는다]
    □ 관련 파일을 직접 열어 읽음: (파일명 기록)
    □ 현재 상태 파악 완료: (확인한 내용 기록)
    □ 이전 단계 부작용 없음 확인
    □ 테스트/빌드 현재 상태 확인

  [실행]
    → 작업 수행

  [완료 후 — 이것을 완료하기 전에는 체크박스를 체크하지 않는다]
    □ 변경된 파일을 다시 열어 1줄씩 읽음: (파일명 기록)
    □ 연결 체인 직접 추적 완료: (추적 경로 기록)
    □ 상위·하위 영향도 확인 완료
    □ 테스트/빌드 100% pass 확인
    □ 이상 없음 → 체크박스 체크
```

## ══ EVIDENCE-BASED VERIFICATION — 증거 기반 검증 규칙 ══

### 코드 연결성 검증 (Micro-Tracing)

```
□ Producer 파일을 열어 출력 필드 목록을 직접 작성한다
□ Consumer 파일을 열어 입력 필드 목록을 직접 작성한다
□ 두 목록을 1:1 대조 → 누락·불일치·미사용 식별한다
□ 중간 변환 단계마다 동일하게 검증한다
□ 직렬화↔역직렬화 경계에서 스키마 일치를 확인한다
□ if/switch 모든 분기 경로에서 필드 접근을 검증한다
```

### 할루시네이션 4대 방어

```
작업 중 아래 4가지를 반드시 파일을 직접 열어 확인한다:

  1. API Shape       : 함수 시그니처 · 파라미터 · 반환값
  2. Export Boundary : barrel파일 · entry point의 실제 공개 항목
  3. Data Flow       : 상수·설정·환경변수의 실제 값·매핑
  4. State Shape     : 인터페이스·클래스·타입 정의의 실제 구조

검증 루프: EXECUTE(실제) → OBSERVE(실제 출력 vs 기대) → CORRECT(근거 기반) → RE-EXECUTE
```

### 거짓 확인 탐지

"확인했다"고 말하려면 아래 중 하나여야 한다:

- 파일을 열어 해당 줄을 직접 본 경우 ✅
- 명령을 실행해 출력 결과를 직접 본 경우 ✅
- 테스트를 실행해 pass 결과를 직접 본 경우 ✅

grep 결과만 본 경우 ❌ | 기억에 의존한 경우 ❌ | "이전에 확인했으니" ❌

## ══ BY WORK TYPE — 작업 유형별 추가 규칙 ══

### 코드 작업 (버그 수정·기능 개발·리팩토링·모듈화)

```
시작 전:
  □ 진입점 전부 열거 (API/Event/Cron/CLI/DI) — 파일 열어 확인
  □ 데이터 흐름 — 진입점→종료점, 모든 분기 끝까지 파일 열어 추적
  □ 모듈 간 의존 관계 — 마이크로 플로우까지 파일 열어 추적
  □ 동적 등록 파악 — Registry/Event Emitter/DI/문자열 디스패치
  □ Barrel/Entry point 공개 API — 파일 열어 직접 확인
  □ VCS clean · 빌드 성공 · 테스트 100% pass · 롤백 경로 확보

실행 원칙:
  SURVEY → PLAN → EXECUTE → TEST → HARDEN → DOCUMENT → VERIFY

  - 증분 방식 기본: 기존 옆에 신규 → 소비자 이주 → 구 코드 삭제
  - Zero Backward Compatibility: 이주 후 구 코드 즉시 삭제
  - Zero Behavioral Change: 구조 변경 시 동작 100% 보존
  - 이상 발견 시 즉시 중단 → 보고 → 지시 대기

완료 후 필수 싱크:
  □ 테스트 import·assertion·fixture → 현행 소스 반영
  □ 타입·인터페이스 변경 → 모든 참조 업데이트
  □ 상수 이동 → 소비자 import 경로 업데이트
  □ 환경 변수 변경 → 테스트 설정·문서 동기화
  □ Mock/Stub → 현행 계약 반영
  □ 고아 테스트 즉시 삭제
```

### 문서 작업 (README·ARCHITECTURE·ADR·주석·CHANGELOG)

```
시작 전:
  □ 기존 문서 파일을 열어 전체 내용 1줄씩 읽기
  □ 문서가 설명하는 코드/구조의 현행 상태 파일 열어 확인
  □ 기존 문서의 오류·구식 내용 목록 작성

실행 원칙:
  - 코드를 먼저 확인한 뒤 문서를 쓴다 (코드가 진실의 소스)
  - 문서에 코드를 복사하면 안 된다 — 코드를 가리키는 참조를 사용한다
  - WHY만 주석으로 남긴다. 코드가 이미 말하는 것은 쓰지 않는다
  - ADR: 2+ 기술 선택지·새 의존성·데이터/API 전략 변경 시 작성

완료 후:
  □ 문서가 현행 코드 상태를 정확히 반영하는지 파일 열어 대조
  □ 문서와 코드 간 불일치 0
```

### 분석·조사 작업 (버그 원인 분석·성능 분석·보안 검토)

```
시작 전:
  □ 분석 대상 전체 범위 명시
  □ 분석 기준선 데이터 확보 (측정값·로그·재현 조건)
  □ 가설을 세우기 전에 데이터를 먼저 수집한다

실행 원칙:
  - 관찰 → 가설 → 검증(파일/실행으로) → 결론 순서를 지킨다
  - 가설은 실제 파일을 열어 검증할 때까지 "가설"이다 — 결론이 아니다
  - 측정 없는 성능 주장 금지 (프로파일링 근거 필수)
  - 보안: 외부 진입점 입력 검증·민감 데이터 암호화·접근 제어 직접 확인

완료 후:
  □ 결론이 실제 증거(파일·측정값·로그)로 뒷받침됨을 명시
  □ "확인된 사실"과 "추론"을 명확히 구분하여 보고
```

### 설계·아키텍처 작업 (새 구조 설계·기술 선택)

```
시작 전:
  □ 현행 구조 전체를 파일 열어 파악 — thoroughly
  □ 변경이 영향을 미칠 모든 모듈 목록 작성
  □ 대안 후보를 최소 2개 준비

실행 원칙:
  - ADR 작성: 문제·선택지·결정·근거·결과
  - 계층 위반·순환 의존성 발생 여부 사전 검토
  - 외부 라이브러리 도입 전: 50줄 이내 직접 구현 가능한가? 먼저 검토

완료 후:
  □ 설계 결정이 현행 코드 구조와 충돌하지 않음을 파일 열어 확인
  □ ADR 작성 완료
```

## ══ SESSION MEMORY PROTOCOL — 세션 메모리 설계 ══

> 세션이 끝나면 컨텍스트는 사라진다. 사라지기 전에 추출하지 않으면 다음 세션은 처음부터 다시 시작한다.
> **세션 종료 전 아래를 반드시 실행한다. 생략 불가.**

### 세션 종료 시 — 메모리 추출 (Session Snapshot)

```
작업 완료 후, 대화 종료 전에 다음을 AGENT_MEMORY.md에 기록한다:

  [SNAPSHOT: YYYY-MM-DD HH:MM]

  ## 현재 상태 (Current State)
  - 작업 중인 브랜치/태스크:
  - 마지막으로 완료된 단계:
  - 다음에 해야 할 것 (정확한 파일명·함수명 포함):
  - 미완료 항목 및 이유:

  ## 핵심 결정 사항 (Key Decisions)
  - 선택한 방향과 이유 (ADR이 없는 경우):
  - 포기한 대안과 이유:

  ## 알려진 위험 (Known Risks)
  - 주의해야 할 파일/모듈:
  - 미검증 가설:
  - 잠재적 충돌 지점:

  ## 컨텍스트 복원용 파일 목록
  - 다음 세션 시작 시 반드시 먼저 열어야 할 파일 (우선순위 순):
    1.
    2.
    3.
```

### 세션 시작 시 — 메모리 복원 (Session Resume)

```
새 세션 시작 즉시, 작업 전에 다음 순서를 따른다:

  1. AGENT_MEMORY.md 파일을 열어 마지막 SNAPSHOT을 읽는다
  2. "컨텍스트 복원용 파일 목록"의 파일을 순서대로 직접 열어 읽는다
  3. "현재 상태"의 미완료 항목을 확인한다
  4. "알려진 위험"을 인지한 상태로 작업을 시작한다

  → AGENT_MEMORY.md가 없으면: UNIVERSAL INVESTIGATION PROTOCOL 전체를 수행한다
  → 5분 이내에 이전 세션의 상태를 복원하는 것이 목표다
```

### 메모리 파일 관리 규칙

```
- AGENT_MEMORY.md는 프로젝트 루트에 위치한다
- 각 스냅샷은 덮어쓰지 않고 append한다 (히스토리 보존)
- 완료된 작업의 스냅샷은 [DONE] 태그로 표시한다
- 파일이 500줄을 초과하면 AGENT_MEMORY_ARCHIVE.md로 이전 스냅샷을 이동한다
```

## ══ PARALLEL AGENT PROTOCOL — 병렬 에이전트 관리 ══

> 에이전트를 병렬로 운용하면 속도는 빠르지만, 충돌 비용은 직렬보다 수배 높다.
> **병렬 작업 시작 전 아래를 반드시 수행한다.**

### 병렬 작업 분해 원칙

```
병렬화 가능 조건 (모두 충족해야 한다):
  □ 각 에이전트의 작업 범위가 파일 단위로 겹치지 않는다
  □ 각 에이전트의 출력이 다른 에이전트의 입력이 되지 않는다 (순차 의존 없음)
  □ 공유 상태(DB·환경변수·설정파일)를 동시에 수정하지 않는다
  □ 각 에이전트의 DoD(완료 정의)가 독립적으로 검증 가능하다

병렬화 금지 조건 (하나라도 해당하면 직렬로 전환):
  ❌ 동일 파일을 2개 이상의 에이전트가 수정
  ❌ 스키마·마이그레이션 변경이 포함된 작업
  ❌ 의존성 설치·삭제가 포함된 작업
  ❌ 이전 에이전트의 완료를 전제로 하는 작업
```

### 에이전트 태스크 카드 (병렬 작업 시 필수)

```
병렬 작업 시작 전, 각 에이전트에 대해 아래를 작성한다:

  [AGENT-{N} TASK CARD]
  - 담당 범위 (파일 목록):
  - 담당 범위 (기능 경계):
  - 완료 정의 (DoD):
  - 접근 금지 파일:
  - 의존하는 선행 에이전트: (없으면 "없음")
  - 충돌 가능 지점:
  - 결과 통합 방법:
```

### 병렬 작업 모니터링

```
에이전트 실행 중:
  □ 각 에이전트의 진행 상태를 AGENT_STATUS.md에 기록한다
     형식: [AGENT-{N}] {상태: RUNNING/DONE/BLOCKED/FAILED} | {마지막 완료 단계} | {다음 단계}

통합(Merge) 시:
  □ 각 에이전트의 DoD 충족 여부를 순서대로 검증한다
  □ 통합 후 전체 빌드·테스트를 반드시 실행한다
  □ 에이전트 간 인터페이스 계약(API shape, 타입) 일치를 파일 열어 직접 확인한다

충돌 발생 시:
  1. 충돌한 두 에이전트를 모두 중단한다
  2. 충돌 전 마지막 안정 상태로 롤백한다
  3. 충돌 원인을 파일 열어 분석한다 — "아마 X 때문일 것이다" 금지
  4. 병렬화 가능 조건을 다시 검토한다
  5. 범위를 재분해하거나 직렬로 전환한다
  6. 재시작 전 AGENT_MEMORY.md에 충돌 원인과 재분해 결정을 기록한다
```

## ══ CODE QUALITY STANDARDS — 코드 품질 기준 ══

코드를 작성·수정하면 아래 기준을 충족해야 한다. 위반 시 완료 선언 불가.

| 항목 | 기준 |
|------|------|
| 순환복잡도 | ≤ 10 |
| 파라미터 수 | ≤ 4 |
| 함수 길이 | ≤ 40줄 |
| 중첩 깊이 | ≤ 3단계 |
| 파일 길이 | ≤200 이상적 · 200~400 수용 · 500+ 분리 |
| `any` / untyped | 0 |
| Magic string/number | 0 (맥락 불명확 시 상수화) |
| 암시적 형변환 | 0 |
| 순환 의존성 | 0 |
| 계층 위반 | 0 |
| Dead reference | 0 |

### 계층 구조
- **Presentation**: 파싱·검증만 (직접 비즈니스 로직 없음)
- **Business**: 순수 도메인 규칙 (외부 의존성 없음)
- **Infrastructure**: 교체 가능한 구현 (DB·API·파일)
- **Cross-cutting**: 미들웨어로 분리 (로깅·인증·모니터링)

### 상수 아키텍처
- Layer1 환경/런타임 · Layer2 도메인 · Layer3 시스템 · Layer4 프레임워크 기본
- 2+ 도메인 공유 → `shared/` 이동 · Barrel로 단일 진입점
- SCREAMING_SNAKE_CASE · 불변 선언 · 상수에서 타입 도출

### 주석 기준
- 허용: `// WHY:` `// INVARIANT:` `// WARNING:` `// HACK(#이슈):` `// CONTEXT:`
- 삭제: 코드가 이미 말하는 것을 반복하는 주석 (노이즈)
- TODO/FIXME: 이슈번호 + 기한 필수 · 기한 없는 TODO 금지

### 리소스 관리
- acquire → use → release 항상 보장
- timer/interval: `clearInterval` / `clearTimeout` cleanup 필수
- process: SIGTERM/SIGINT 핸들러 · graceful shutdown
- 에러: 삼키기 금지 · WHAT+WHY+WHERE+WHEN 포함

## ══ TASTE & JUDGMENT — 설계 감각과 판단 기준 ══

> LLM이 코드를 작성하는 시대일수록, "무엇을 만들지 말지"를 판단하는 감각이 결과를 가른다.
> 아래는 규칙이 아니라 판단 기준이다. 체크리스트처럼 통과하는 것이 목적이 아니라,
> 진짜로 더 나은 결정을 내리기 위한 질문들이다.

### 만들기 전에 물어야 할 것

```
□ 이게 정말 필요한가?
  → 지금 당장 실제로 발생하는 문제인가? 미래 예상 문제는 만들지 않는다.
  → "나중에 필요할 것 같아서"는 이유가 되지 않는다.

□ 더 단순한 방법이 있는가?
  → 이 기능을 50줄로 줄일 수 있는가?
  → 외부 라이브러리 없이 직접 구현 가능한가?
  → 기존 코드를 재사용하면 안 되는 이유가 있는가?

□ 이 추상화가 실제로 복잡성을 줄이는가?
  → 추상화 레이어를 추가하면 이해해야 할 것이 늘어난다.
  → 새 레이어가 제거하는 복잡성 > 추가하는 복잡성일 때만 만든다.
  → "우아해 보여서"는 이유가 되지 않는다.
```

### 코드 리뷰 시 물어야 할 것

```
□ 6개월 후의 나(또는 동료)가 이 코드를 처음 보면 이해할 수 있는가?
□ 이 함수의 이름이 실제로 하는 일을 정확히 설명하는가?
□ 이 테스트가 실제 사용 시나리오를 커버하는가, 아니면 구현 디테일을 테스트하는가?
□ 이 에러 핸들링이 실제 장애 상황에서 디버깅에 도움이 되는가?
```

### 에이전트에게 위임 전에 물어야 할 것

```
□ 내가 직접 했을 때와 비교해 결과 품질 기준이 동일한가?
  → 에이전트가 했기 때문에 기준을 낮추는 것은 기술 부채를 쌓는 것이다.

□ 이 작업의 DoD(완료 정의)를 내가 명확하게 말할 수 있는가?
  → 말할 수 없으면 에이전트도 모른다.

□ 에이전트 결과를 내가 검증할 수 있는가?
  → 검증 능력이 없는 영역을 에이전트에게 위임하면 결과를 신뢰할 수 없다.
  → 위임 전에 해당 영역에 대한 최소한의 이해를 갖춘다.

□ 이 작업이 잘못됐을 때 롤백 가능한가?
  → 롤백 불가능한 작업(프로덕션 DB 수정 등)은 에이전트에게 단독 위임하지 않는다.
```

### 하지 말아야 할 것 (Anti-Patterns)

```
❌ 에이전트 결과를 읽지 않고 "됐겠지"로 머지
   → 에이전트의 "완료"는 나의 "완료"가 아니다

❌ 복잡한 문제를 프롬프트 길이로 해결하려는 시도
   → 문제가 복잡하면 분해가 먼저다 — 프롬프트 길이가 아니다

❌ 같은 방식으로 실패한 작업을 같은 방식으로 재시도
   → 실패 유형을 먼저 분석한다: 컨텍스트 부족 / 범위 불명확 / 기술적 한계

❌ 에이전트가 만든 코드라서 리팩토링을 미루는 것
   → 코드 품질 기준은 작성 주체와 무관하다

❌ taste 없이 속도만 추구
   → 에이전트 10개를 동시에 돌려도 방향이 틀리면 10배 빠르게 잘못된 곳에 도착한다
```

## ══ MODULIZATION RULES — 모듈화 전용 규칙 ══

```
NEVER                                    ALWAYS
──────────────────────────────────────────────────────────────
전체 시스템 한 번에 모듈화               전수조사 완료 후 시작 (thoroughly)
계획서 없이 착수                          딱 하나, 가장 급한 것부터
구 코드 삭제 안 하고 공존               계획서 작성 → 체크하며 진행
기존 동작 변경하며 모듈화               Zero Backward Compatibility — 구코드 즉시 삭제
grep으로 "확인"                           Zero Behavioral Change — 동작 100% 보존
테스트 없이 이주 완료 선언             각 이주 단계 후 테스트 100% pass
```

**이주 순서:**
```
1. 기존 코드에 특성화 테스트 작성 (현행 동작 캡처, 버그 포함)
2. 새 모듈 구조 생성 (기존 옆에 신규 구현)
3. 소비자 이주 — 한 번에 하나씩
   ├─ 테스트 import·assertion·fixture → 현행 경로/시그니처 업데이트
   ├─ 타입·인터페이스 변경 → 모든 참조 업데이트
   └─ 상수 이동 → 소비자 import 경로 업데이트
4. 각 단계 후 테스트 100% pass 확인
5. 구 구현 완전 삭제 + 고아 테스트·타입·상수 즉시 삭제
6. POST 전수조사 3회 실행
```

## ══ POST-WORK MANDATORY AUDIT — 작업 완료 후 필수 전수조사 ══

> ⛔ **모든 작업 완료 후 아래 전수조사를 3회 실행한다. 생략 불가. 한 회도 건너뛰지 마라.**
> grep/패턴 매칭으로 "확인했다" 판단 ❌ — 파일을 직접 열어 1줄씩 추적만 인정.
> 변경사항이 많을수록 더 중요하다. thoroughly 수행하라.

```
【전수조사 체크리스트 — 3회 반복, 매 회 파일 직접 열어 추적】

▸ 안전성 (Safety)
  □ 삭제된 코드를 참조하는 지점 0 — 관련 파일 열어 직접 확인
  □ 새 구조로의 이주가 누락된 소비자 0 — 소비자 파일 전부 열어 확인
  □ 빌드 0에러 · 정적 분석 0에러 — 실행 결과로 확인
  □ 전체 테스트 100% pass, regression 0 — 실행 결과로 확인

▸ 연결성 (Connectivity) — 파일 직접 열어 1줄씩 추적. grep 절대 금지.
  □ 모든 import/export 경로 — 파일 열어 직접 추적 완료
  □ 동적 연결(Registry·이벤트·DI·문자열 디스패치) — 파일 열어 추적 완료
  □ Barrel/Entry point 공개 API — 파일 열어 export 목록 직접 확인
  □ Producer→Consumer 필드 1:1 매칭 — 양쪽 파일 열어 목록 직접 대조
  □ 고아 코드(와이어업 누락) 0 — 파일 열어 직접 확인

▸ 일관성 (Consistency)
  □ 새 모듈·파일 명명 규칙 전체 통일 — 파일명 목록 직접 확인
  □ 계층 구조 일관성 — Presentation/Business/Infrastructure 파일 열어 확인
  □ 상수·타입 참조 → 현행 정의를 가리킴 — 정의 파일 열어 직접 확인
  □ 문서(README/ARCHITECTURE/CHANGELOG) 현행 구조 반영 — 파일 열어 확인

▸ 전체 싱크 (Full Sync) — 소스 변경으로 영향받은 모든 것
  □ 테스트 코드 — 시그니처·임포트·어설션·fixture 파일 열어 현행 소스와 대조
  □ 고아 테스트 0 — 삭제된 소스 대응 테스트 삭제 확인
  □ 누락 테스트 0 — 새 공개 함수에 테스트 존재 확인
  □ 타입 정의 변경 — 모든 참조 파일 열어 업데이트 확인
  □ 상수/설정 이동·이름 변경 — 모든 소비자 파일 열어 경로 업데이트 확인
  □ 환경 변수 변경 — 테스트 설정·문서 파일 열어 싱크 확인
  □ Mock/Stub — 현행 계약(시그니처·반환값) 파일 열어 확인
  □ 문서(README·ARCHITECTURE·CHANGELOG·ADR) — 파일 열어 현행 구조 반영 확인

▸ 프로젝트 영향도 분석 (thoroughly)
  □ 변경 모듈의 상위 의존성 — 모든 소비자 파일 열어 추적
  □ 변경 모듈의 하위 의존성 — 모든 의존 파일 열어 추적
  □ 작은 의존관계부터 마이크로 플로우까지 끝까지 — thoroughly
  □ 문제 없음 확인. 문제 발견 시 즉시 보고 후 수정.

▸ 코드 품질 최종 확인 (코드 작업 시)
  □ Dead reference 0 · 고아 코드 0 · Dead code 0
  □ Micro-Tracing으로 검증 완료 (grep 아님) · Producer→Consumer 1:1 매칭
  □ 순환 의존성 0 · 계층 위반 0 · 외부 의존성 어댑터 격리
  □ Magic value 0 · 순환복잡도 ≤10 · 중첩 ≤3 · 파라미터 ≤4 · any 0
  □ 노이즈 주석 0 · WHY 주석 존재 · TODO에 이슈번호+기한
  □ 공개 함수별 테스트 존재 · 경계값+실패경로 커버리지
  □ 테스트 import→현행모듈 · 함수호출→현행시그니처 · 어설션→현행동작
  □ API shape·Export boundary·Data flow·State shape 할루시네이션 0

▸ 세션 마무리 (Session Wrap-up)
  □ AGENT_MEMORY.md에 현재 세션 스냅샷 기록 완료
  □ 병렬 에이전트 운용 중이었다면 AGENT_STATUS.md 업데이트 완료
  □ 미완료 항목과 다음 세션의 시작 지점 명시 완료
```

> ✅ 위 체크리스트를 **3회 전부** 통과한 뒤에만 작업을 완료로 선언할 수 있다.

## ══ COMPLETION DECLARATION — 완료 선언 조건 ══

아래를 모두 증거와 함께 제시해야 완료를 선언할 수 있다:

```
□ 변경된 파일 전체를 직접 열어 1줄씩 읽었다 — (파일 목록 명시)
□ 실행한 명령어: (명시)
□ 결과: X passed / Y failed / 빌드 성공 (실제 출력 명시)
□ 부작용 검토 완료 — (검토 내용 명시)
□ 영향받은 모든 것 싱크 완료 — (테스트·타입·상수·문서 각각 명시)
□ Post-Work 전수조사 3회 완료 — thoroughly
□ Dead reference 0 — 직접 추적한 경로 명시
□ Producer→Consumer 1:1 매칭 확인 — 직접 추적한 경로 명시
□ 테스트 고아·누락 0 — 직접 확인 내용 명시
□ AGENT_MEMORY.md 세션 스냅샷 기록 완료
```

> ✅ 위 항목 전부를 증거와 함께 제시하지 않으면 완료를 선언할 수 없다.
